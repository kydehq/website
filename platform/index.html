What KYDE WITNESS is

Not a filter.
A cryptographic evidence layer.

KYDE WITNESS does not inspect, block, or rate-limit your prompts by default. It is a transparent proxy that records what your AI agents did, in a form that cannot be altered after the fact.

Think of it as the black box for your agent fleet. It runs silently, adds less than 100ms of latency, and produces a tamper-evident audit trail that survives any audit, any incident investigation, and any regulatory inquiry.

Architecture

Three layers. One proxy.

The Proxy

KYDE WITNESS runs as a local or hosted HTTP proxy, fully compatible with the OpenAI API schema. Any agent that can be pointed at a base URL can be governed.

OPENAI_BASE_URL=http://localhost:8080/v1

One environment variable. No SDK dependency. No code changes required.

The Append-Only Database

Every LLM interaction produces a signed, hash-chained entry. Tamper with any single entry, and every subsequent link breaks.

  • Who acted agent ID, machine ID, user identity
  • What happened prompt, model, tool calls, response
  • Why it happened last 5 messages as causal context
  • When millisecond-precise UTC timestamp
  • Proof Ed25519 sig + SHA-256 hash chain

The Signing Layer

Each entry is signed with an Ed25519 keypair generated locally on first initialization.

In Phase 3, signing is optionally rooted in a PKCS#11 hardware security module (HSM) or Trusted Execution Environment (TEE) for the highest levels of evidential integrity.

Ed25519 PKCS#11 HSM TDX/SEV-SNP

The Three Phases

Start with visibility. Progress to control.

Phase 1 Available Now

AUDIT

Prove what happened

Passive observation and evidence preservation. The proxy intercepts every LLM call, signs it, and chains it. Nothing is blocked. Nothing is altered. You gain complete forensic visibility with zero operational risk.

Deployment: ENV variable or Group Policy push. No developer involvement required.

  • Tamper-evident append-only database
  • Ed25519 signing + SHA-256 hash chaining
  • Causal context capture (the "Why")
  • CLI audit and verification tools
  • Provider-agnostic interception
Phase 2 Coming Soon

GOVERN

Control what happens

The first active intervention layer. WITNESS starts enforcing economic and behavioral boundaries — without needing any developer code changes.

  • Token count and API spend limits
  • API allowlists and blocklists per agent
  • Real-time behavioral anomaly detection
  • Continuous chain integrity verification
  • Alerting for out-of-policy behavior
Phase 3 Roadmap 2027

ATTEST

Govern who acts

The full trust layer. Semantic policy enforcement, hardware-rooted signing, and machine-readable delegation — enabling agents to prove their authority to any counterparty.

  • Policy Enforcement Point with circuit breakers
  • HSM / TEE-rooted signing
  • Machine-readable Power of Attorney
  • Agent-to-agent delegation model

Deployment

Three ways to deploy. Zero code changes.

01

Environment Variable

Developer / Single machine

export OPENAI_BASE_URL=http://localhost:8080/v1

Point any agent at the proxy with one ENV variable. Ideal for local development or single-machine deployments.

02

Group Policy

IT / Enterprise fleet

Push a single GPO that redirects all LLM API traffic through KYDE WITNESS. No developer action, no application changes, no opt-in required.

03

Hosted Proxy

Cloud / SaaS

For teams that prefer a managed deployment. Traffic routes through KYDE's infrastructure. Data-residency options available for EU-regulated environments.
